## 三、推荐开发顺序

### 阶段一：基础设施（P0）—— 必须最先完成

| 顺序 | 模块 | 说明 |
|:----:|------|------|
| 1 | `init-project` | 搭建前后端脚手架和 Docker 环境 |
| 2 | `init-database` | 建表、插初始数据 |
| 3 | `common` | 公共类（Result、异常、枚举、工具类） |
| 4 | `auth` | 登录认证、JWT、前端布局框架 |

> **阶段一完成后**：系统可以登录，能看到空的管理后台布局。

### 阶段二：核心业务（P1）—— 可部分并行

| 顺序 | 模块 | 说明 | 可并行？ |
|:----:|------|------|:--------:|
| 5 | `user` | 用户与组织、数据权限基础 | 与 notification 并行 |
| 6 | `notification` | 通知服务 + WebSocket | 与 user 并行 |
| 7 | `plan` | **核心**：计划 CRUD、提交、日历、汇总 | 依赖 user |
| 8 | `achievement` | 成果 CRUD、提交、对比 | 依赖 plan |
| 9 | `approval` | 审批流转（通过/驳回/转审） | 依赖 plan + achievement + notification |

> **阶段二完成后**：核心业务闭环跑通——员工报计划→领导审批→员工提交成果→领导验收。

### 阶段三：辅助功能（P2）

| 顺序 | 模块 | 说明 | 可并行？ |
|:----:|------|------|:--------:|
| 10 | `file` | 附件上传下载预览 | 与 statistics 并行 |
| 11 | `statistics` | 统计图表 | 与 file 并行 |
| 12 | `admin` | 管理后台页面 | 依赖 user + approval |

> **阶段三完成后**：系统功能完整，可用。

### 阶段四：部署上线（P3）

| 顺序 | 模块 | 说明 |
|:----:|------|------|
| 13 | `deploy` | Docker 镜像构建、Compose 编排、一键部署 |

> **阶段四完成后**：系统可 Docker 一键部署运行。

## 六、使用说明

1. 开发时，按模块对应的 `tasks/<module-name>.md` 文件逐个勾选完成的任务
2. 本文件中的模块完成状态根据对应任务文件的全部任务是否完成来更新
3. 阶段内模块开发完成且功能可用后，将模块状态改为 `✅ 已完成`
4. 遇到阻塞时，在对应模块文件底部添加 `## 阻塞事项` 记录
