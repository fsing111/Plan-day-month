# 日/月计划和成果验收系统 - 需求文档

## 一、项目概述

### 1.1 项目背景

部门内部需要一套工作计划管理和成果验收系统，用于规范员工的日常工作汇报流程。员工每日、每周、每月提交工作计划，完成后提交成果；领导逐级审批，实现工作过程的可视化和可追溯。

### 1.2 适用场景

- **组织规模**：单个部门/团队，约 10-30 人
- **使用频率**：每日使用，贯穿日常工作流程

### 1.3 核心目标

1. 员工能便捷地填写日/周/月计划并提交成果
2. 领导能高效地审批计划和验收成果
3. 计划和成果可查询、可追溯、可统计
4. 日报 → 周报 → 月报形成逐级汇总，方便回顾总结

## 二、用户角色

| 角色 | 说明 | 核心权限 |
|------|------|----------|
| **员工** | 普通团队成员 | 填写个人计划、提交成果、查看自己的计划和成果 |
| **领导** | 直属上级/部门负责人 | 审批下属计划、验收下属成果、查看下属的计划和成果、管理部门人员 |
| **系统管理员** | 由领导兼任 | 用户管理、组织架构设置、审批链配置 |

### 2.1 权限矩阵

| 功能 | 员工 | 领导 | 管理员 |
|------|:---:|:---:|:-----:|

## 三、功能需求

### 3.1 计划管理

#### 3.1.1 计划层级

系统支持三个层级的计划，层级之间有关联汇总关系：

```
月报（月度计划）
  └── 周报（周度计划）—— 汇总本周的日报内容
        └── 日报（每日计划）—— 汇总到所属周报
```

- **日报汇总到周报**：填写周报时，可自动引用/汇总本周的日报内容
- **周报汇总到月报**：填写月报时，可自动引用/汇总本月的周报内容

#### 3.1.2 计划字段

每个计划（日/周/月通用）包含以下字段：

| 字段 | 类型 | 必填 | 说明 |
|------|------|:---:|------|
| 计划标题 | 文本 | ✅ | 简洁概括计划内容 |
| 详细描述 | 富文本 | ❌ | 计划的详细说明 |
| 计划类型 | 枚举 | ✅ | 日报 / 周报 / 月报 |
| 开始时间 | 日期时间 | ✅ | 计划开始时间 |
| 结束时间 | 日期时间 | ✅ | 计划截止时间 |
| 优先级 | 枚举 | ✅ | 高 / 中 / 低 |
| 所属项目/分类 | 枚举 | ❌ | 自定义分类，如"项目A"、"日常运维"等 |
| 量化指标 | 文本 | ❌ | 预计完成的具体数量或指标 |

#### 3.1.3 计划状态流转

```
草稿 → 已提交 → 审批中 → 已通过 / 已驳回
                              ↓
                        （驳回后可修改重新提交）
```

#### 3.1.4 时间约束（建议规则）

| 计划类型 | 提交时间要求 |
|----------|------------|
| 日报 | 当日早上 9:00 前提交当日计划；或前一天下班前提交次日计划 |
| 周报 | 每周一早上 9:00 前提交本周计划；或上周五下班前提交 |
| 月报 | 每月 1 号早上 9:00 前提交本月计划；或上月最后一个工作日提交 |

> 以上时间规则为建议性约束，系统通过提醒功能督促，不强制阻止超时提交。

### 3.2 成果管理

#### 3.2.1 成果提交

员工完成计划后，提交成果进行验收。成果与计划一一对应。

#### 3.2.2 成果字段

| 字段 | 类型 | 必填 | 说明 |
|------|------|:---:|------|
| 关联计划 | 关联 | ✅ | 对应哪条计划 |
| 完成说明 | 富文本 | ✅ | 描述实际完成情况 |
| 实际完成数量 | 数字 | ❌ | 与计划中的量化指标对比 |
| 实际耗时 | 数字 | ❌ | 实际花费时间（小时） |
| 附件 | 文件 | ❌ | 截图、文档等成果佐证材料（支持多个附件） |
| 遇到的问题 | 文本 | ❌ | 过程中遇到的问题或阻塞 |
| 备注 | 文本 | ❌ | 其他补充说明 |

#### 3.2.3 成果状态流转

```
待填写 → 已提交 → 验收中 → 已通过 / 已驳回
                            ↓
                      （驳回后可修改重新提交）
```

#### 3.2.4 量化对比

领导验收时，系统自动展示**计划 vs 实际**的对比：

- 计划指标 vs 实际完成数量
- 预计耗时 vs 实际耗时
- 偏差一目了然，辅助领导判断

---

### 3.3 审批流程

#### 3.3.1 审批链设计

支持多级审批，审批链可配置。默认支持两级：

```
员工提交 → 直属领导（一级审批）→ 部门负责人（二级审批/终审）
```

每个审批节点可执行操作：

| 操作 | 说明 |
|------|------|
| **通过** | 同意该计划/成果，进入下一级审批或完成 |
| **驳回** | 退回给提交人，需填写驳回原因 |
| **转审** | 转给其他人审批（如原审批人不在） |

#### 3.3.2 审批记录

每次审批操作记录：
- 审批人
- 审批时间
- 审批结果（通过/驳回/转审）
- 审批意见（文字）

#### 3.3.3 可扩展性

审批链级数设计为可配置，支持未来扩展为 3 级、4 级审批。

---

### 3.4 查询与展示

#### 3.4.1 列表视图

支持多条件筛选：
- 按**计划类型**筛选（日/周/月）
- 按**状态**筛选（草稿/审批中/已通过/已驳回）
- 按**时间范围**筛选
- 按**优先级**筛选
- 按**项目/分类**筛选
- 按**提交人**筛选（领导可见下属）
- 支持**关键词搜索**

#### 3.4.2 日历视图

- 以日历形式展示每日的计划安排
- 日视图 / 周视图 / 月视图 可切换
- 日历上不同颜色标识不同状态（已通过/待审批/驳回等）
- 点击日期可查看当天详细计划

#### 3.4.3 统计分析

- **个人统计**：我的计划完成率、按时提交率、审批通过率
- **团队统计**（领导可见）：团队整体完成率、个人排名、未提交人员
- **趋势图**：近期完成率变化趋势
- **可视化图表**：饼图（状态分布）、柱状图（完成率对比）、折线图（趋势）

---

### 3.5 消息通知

#### 3.5.1 通知场景

| 触发事件 | 通知对象 | 通知方式 |
|----------|----------|----------|
| 计划/成果被驳回 | 提交人 | 系统内通知 |
| 计划/成果通过终审 | 提交人 | 系统内通知 |
| 有新的待审批项 | 审批人 | 系统内通知 |
| 日报/周报/月报未按时提交 | 未提交的员工 | 系统内提醒 |
| 审批超时未处理 | 审批人 | 系统内提醒 |

#### 3.5.2 通知方式

- **系统内通知**：右上角通知铃铛 + 未读红点 + 通知列表
- 后续可扩展邮件通知

---

### 3.6 附件管理

- 成果提交时支持上传附件（图片、PDF、Word、Excel 等常见格式）
- 单个文件大小限制：10MB
- 单次提交附件数量限制：5 个
- 附件预览：图片支持在线预览，其他文件提供下载
- 附件存储在服务器文件系统或对象存储中

---

### 3.7 系统管理

由领导兼任的管理员负责：

| 功能 | 说明 |
|------|------|
| **用户管理** | 添加/编辑/禁用用户账号，设置员工基本信息 |
| **部门管理** | 设置部门结构 |
| **上下级关系** | 配置员工的直属领导，建立汇报关系 |
| **审批链配置** | 设置审批级数和各级审批人 |
| **项目/分类管理** | 维护计划所属的项目或分类标签 |

## 四、非功能需求

### 4.1 性能要求

- 页面加载时间 < 3 秒
- 列表查询响应时间 < 1 秒
- 支持 30 人同时在线操作

### 4.2 安全要求

- 用户登录认证（JWT Token）
- 基于角色的权限控制（RBAC）
- 数据权限隔离（员工只能看自己的数据）
- 密码加密存储
- API 接口鉴权

### 4.3 可用性

- 界面简洁直观，降低学习成本
- 移动端响应式适配（方便手机上填写日报）
- 关键操作有确认提示，防止误操作

### 4.4 可维护性

- 前后端分离，独立部署
- Docker 容器化部署，方便环境迁移
- 代码结构清晰，注释完整

## 六、核心数据模型（概要）

### 6.1 实体关系

```
用户(User) ──1:N──> 计划(Plan) ──1:1──> 成果(Achievement)
  │                    │
  │                    ├── 审批记录(ApprovalRecord)
  │                    │
  │                    └── 附件(Attachment)
  │
  └── 部门(Department)
        │
        └── 审批链配置(ApprovalChain)
```

### 6.2 核心表

| 表名 | 说明 |
|------|------|
| `user` | 用户表（员工/领导/管理员） |
| `department` | 部门表 |
| `plan` | 计划表（日/周/月计划） |
| `plan_weekly_ref` | 日报-周报关联表 |
| `plan_monthly_ref` | 周报-月报关联表 |
| `achievement` | 成果表 |
| `attachment` | 附件表 |
| `approval_record` | 审批记录表 |
| `approval_chain` | 审批链配置表 |
| `notification` | 通知表 |
| `project_category` | 项目/分类表 |

## 七、待确认事项

以下事项建议在详细设计阶段进一步明确：

1. **审批提醒的时间阈值**：提交后多久未审批触发提醒？
2. **计划提交的严格程度**：超时是否允许提交？是否需要补交机制？
3. **历史数据归档**：是否需要将历史计划和成果归档？
4. **邮件通知**：是否需要接入邮件服务？使用什么邮件服务？
5. **移动端**：需要专门的移动端应用还是浏览器响应式适配即可？
6. **审批链配置粒度**：按部门配置还是可以按个人配置？
